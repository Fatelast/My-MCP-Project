# 技术方案评审报告

## 摘要

- **评审结论**：✅ 有条件通过（核心架构合理，需关注役种引擎复杂度）
- **主要风险**：
  1. 役种判定算法复杂，边界用例多，需充分测试

## 5. 架构改进建议

### 5.1 必须修改（阻塞项）

- [ ] **AI 超时兜底**：AI 决策必须有 2000ms 超时，超时后自动出安全牌，防止游戏卡死
- [ ] **立直后出牌限制**：立直后玩家只能出摸到的牌（除非暗杠），需在 UI 层强制约束

### 5.2 建议优化（非阻塞）

- [ ] **役种引擎 TDD**：先写测试用例再实现，节省后期 debug 时间
- [ ] **Zustand immer 中间件**：状态更新使用 immer，避免手动展开 spread
- [ ] **AI 在 Web Worker 中运行**：防止复杂计算阻塞 UI 线程
- [ ] **错误边界（ErrorBoundary）**：防止游戏逻辑异常导致整个页面崩溃

## 7. 代码规范

### 7.1 命名规范

- **文件**：camelCase（`gameStore.ts`）或 PascalCase（`GameBoard.tsx`）
- **组件**：PascalCase（`PlayerArea`）
- **函数**：camelCase（`calculateScore`）
- **常量**：UPPER_SNAKE_CASE（`MAX_HAND_SIZE`）
- **类型/接口**：PascalCase（`GameState`, `Tile`）

### 7.2 代码风格

- TypeScript strict 模式，禁止 `any`
- 游戏引擎函数必须是纯函数（相同输入相同输出）
- 每个公共函数必须有 JSDoc 注释
- Airbnb ESLint 规则
- Commit 遵循 Conventional Commits
